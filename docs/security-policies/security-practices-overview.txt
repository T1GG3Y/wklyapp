## Identity and Access Management

### Question 3: Access Controls

**Answer:**
- A defined and documented access control policy is in place
- Periodic access reviews and audits are performed

**Documentation:** See `access-controls-policy.md`

**Summary:**
- Role-Based Access Control (RBAC) implemented
- Least privilege principle enforced
- Quarterly access reviews for all users
- Monthly reviews for privileged accounts
- Access logging and monitoring enabled

### Question 4: Consumer MFA (Before Plaid Link)

**Answer:** Yes - Non-phishing-resistant multi-factor authentication is performed (e.g., SMS, email)

**Implementation:**
- Consumers authenticate with email/password before accessing Plaid Link
- MFA options include:
  - SMS verification codes
  - Email verification codes
- MFA triggered for:
  - New device logins
  - Account recovery
  - Sensitive actions

### Question 5: MFA for Critical Systems (Consumer Financial Data)

**Answer:** Yes - MFA is required for all access to critical systems

**Implementation:**
- All production system access requires MFA
- Cloud console access (Vercel, Firebase/Google Cloud) requires MFA
- GitHub repository access requires MFA
- Database access requires MFA
- MFA methods: Authenticator apps, hardware keys

## Infrastructure and Network Security

### Question 6: TLS 1.2+ Encryption in Transit

**Answer:** Yes

**Implementation:**
- All client-server communication uses TLS 1.2 or higher
- HTTPS enforced on all endpoints (no HTTP fallback)
- Vercel provides automatic TLS certificate management
- Firebase/Firestore connections use TLS encryption
- API endpoints enforce TLS 1.2 minimum

**Technical Details:**
- Frontend hosted on Vercel with automatic HTTPS
- Firebase/Firestore connections use SSL/TLS
- Firebase Authentication uses secure HTTPS endpoints
- Plaid API communication uses TLS 1.3
- HSTS headers enabled

### Question 7: Encryption at Rest

**Answer:** Yes

**Implementation:**
- Database encryption: Firebase/Firestore uses AES-256 encryption at rest by default
- All consumer financial data encrypted at the database level
- Backup encryption enabled
- Encryption keys managed by Google Cloud with SOC 2 compliance

**Technical Details:**
- Firestore: Data encrypted at rest using AES-256 (Google default encryption)
- Firebase Authentication: Credentials securely hashed and stored
- Vercel: No persistent consumer data stored
- API keys and secrets: Encrypted in environment variables

## Development and Vulnerability Management

### Question 8: Vulnerability Scanning

**Answer:** Yes - We perform vulnerability scanning

**Practices in Place:**
- Dependency vulnerability scanning (npm audit, Dependabot)
- Code security scanning in CI/CD pipeline
- Regular review of security advisories
- Prompt patching of identified vulnerabilities

**Details:**
- GitHub Dependabot alerts enabled for dependency vulnerabilities
- npm audit run as part of build process
- Critical vulnerabilities addressed within 24-48 hours
- High vulnerabilities addressed within 1 week

### Question 10: Consumer Consent

**Answer:** Yes - We obtain consent from consumers

**Documentation:** See `consumer-consent-procedures.md`

**Implementation:**
- Consent obtained at account registration
- Explicit consent before Plaid Link connection
- Clear disclosure of data collection and usage

**Consent Points:**
1. Account registration: Privacy Policy acceptance required
2. Financial account connection: Pre-Link disclosure + Plaid consent flow
3. Marketing (optional): Separate opt-in consent

**Documentation:** See `data-retention-deletion-policy.md`

**Summary:**
- Retention periods defined by data category

- Policy reviewed semi-annually
